# City Defense: drone attack vs mobile road-based defenders

A multi-agent RL prototype of 10 attacking drones trying to reach a city, and
5 defender groups that move **only on a road network** to intercept them.

The prototype uses a **hierarchical** setup with three policies:

| policy | role | cadence | action type |
|---|---|---|---|
| **Attacker planner** | picks the full attack plan for all 10 drones | once, at episode start | structured discrete (per drone: launch time, entry point, route template, speed mode) |
| **Defender deployment** | picks initial defender positions on the road graph | once, at episode start | 5 discrete road-node choices |
| **Defender tactical** | moves defender groups along roads | every few seconds during the episode | per-defender next-node choice |

Scripted baselines exist for every policy and serve as opponents during
phased PPO training (via [Stable-Baselines3](https://github.com/DLR-RM/stable-baselines3)).

## Training (phased)

Follow the four phases below; each uses the previously trained models as
(scripted or frozen) opponents.

### Phase 1 — sanity check
```bash
.venv/bin/python evaluation/eval.py --episodes 20
```
Scripted vs scripted. Defender typically wins ~65-75% of rounds.

### Phase 2 — tactical defender
```bash
.venv/bin/python training/train_tactical_defender.py \
    --timesteps 200000 --n_envs 4 \
    --out outputs/models/tactical_defender \
    --log outputs/plots/tactical_training.csv
```
Uses scripted attacker + scripted deployment. Evaluate with:
```bash
.venv/bin/python evaluation/eval.py --episodes 20 \
    --tactical_model outputs/models/tactical_defender
```

### Phase 3 — attacker planner
```bash
.venv/bin/python training/train_attacker.py \
    --timesteps 60000 --n_envs 4 \
    --tactical_model outputs/models/tactical_defender \
    --out outputs/models/attacker_planner
```
Each RL step commits a full plan and runs a complete episode, so 60k steps =
60k episodes.

### Phase 4 — deployment policy
```bash
.venv/bin/python training/train_deployment.py \
    --timesteps 40000 --n_envs 4 \
    --tactical_model outputs/models/tactical_defender \
    --out outputs/models/deployment
```

### (Optional) Phase 5 — alternating self-play
```bash
.venv/bin/python training/self_play.py \
    --iterations 2 --tactical_steps 50000 --attacker_steps 15000
```

## Evaluation

`evaluation/eval.py` runs *any* mix of scripted and trained policies:

```bash
.venv/bin/python evaluation/eval.py --episodes 20 \
    --attacker_model outputs/models/attacker_planner \
    --deployment_model outputs/models/deployment \
    --tactical_model outputs/models/tactical_defender \
    --save_first_replay
```

Prints mean reached / destroyed / defender win rate; optionally saves a
replay plot for the first episode.

## What the environment looks like

### Fixed map
- Unit square `[0,1] x [0,1]`.
- City circle at `(0.5, 0.5)`, radius `0.126` (~5% of area).
- Hand-designed road graph: **28 nodes**, ~70 edges, **8 entry points** on the map boundary.
- 20 deployment-candidate nodes (inner + mid ring around the city).

### Time
- `dt = 1 s`, `T = 180 steps` per episode.
- Tactical defender decides every 4 steps; the engine advances physics between decisions.

### Drones
- Point mass with max speed `0.020` and max acceleration `0.010` (units per second).
- Trajectory is generated by tracking waypoints from the chosen **route template** (direct / left flank / right flank / wide loop / late turn / feint).
- Speed mode multiplies max speed by `{0.6, 1.0, 1.3}`.
- A drone is **destroyed** when its distance to any defender is `<= defender_intercept_radius = 0.035`.
- A drone **reaches the city** when it enters the city circle.

### Defenders
- Each defender lives on a road edge, moving with speed `0.012`.
- At tactical decisions the policy picks a slot from the defender's candidate-neighbor list (slot 0 = "stay"); if a non-adjacent target is selected indirectly (via scripted tactical), the engine routes via shortest path.
- Interception is a pure 2D distance check.

### Observations

| policy | what it sees |
|---|---|
| Attacker | city center + radius, node positions, entry points, defender start positions (each visible with prob 0.5, otherwise masked to `-1`). |
| Deployment | city center + radius, full node positions, allowed deployment nodes. |
| Tactical | time, per-drone (pos, vel, alive/launched), per-defender pos, city center (full observability of drones for the prototype). |

### Action spaces

| policy | action |
|---|---|
| Attacker | `MultiDiscrete([12, 8, 6, 3] * 10)` = per drone `(launch_bin, entry_point_id, route_template_id, speed_mode)`. Launch bins map to `[0, 0.7T]` uniformly. |
| Deployment | `MultiDiscrete([20] * 5)` = an index into the 20 deployment-candidate nodes, per defender. |
| Tactical | `MultiDiscrete([6] * 5)` = per defender, slot into its `[stay, nbr1, ..., nbr5]` list. |

### Rewards

| event | attacker | defender |
|---|---|---|
| drone reaches the city | `+10` | `-10` |
| drone destroyed | `-5` | `+5` |

Zero-sum-ish by construction. Defender reward is **shared** between deployment and tactical policies for the prototype (each gets the full episode reward).

## Project layout

```
envs/
  fixed_map.py            # hand-designed map: graph, city, entry points
  route_templates.py      # 6 predefined route families (direct, flanks, ...)
  dynamics.py             # drone point-mass + defender-on-graph motion
  city_defense_env.py     # core engine with 3 commit points (attacker/deploy/tactical)
  wrappers.py             # 3 gymnasium wrappers for SB3 (one per policy)
agents/
  scripted_attacker.py    # randomised-but-sensible attack plans
  scripted_defender_deployment.py   # high-degree + angular-coverage heuristic
  scripted_defender_tactical.py     # greedy nearest-threat interception
training/
  train_tactical_defender.py
  train_attacker.py
  train_deployment.py
  self_play.py            # small alternating loop (prototype)
  common.py               # config loader, VecEnv helper, CSV stats callback
evaluation/
  eval.py                 # mix any combination of scripted/trained policies
visualization/
  render.py               # static plots + animated gif/mp4
scripts/
  demo_scripted.py        # one-liner end-to-end run
configs/
  default.yaml            # all tunable env + training hyperparameters
outputs/
  plots/                  # map overview, training curves
  replays/                # per-episode replay images + gifs
  models/                 # SB3 .zip checkpoints
```

## Deliberate non-goals

This is a prototype, not a simulator. The following are **out of scope**:

- arbitrary continuous attacker trajectories (we use a small discrete route library)
- partial observability for the tactical defender (it sees all drones)
- noisy sensors, comms delay, decentralized MARL
- random maps, military realism

---

## Extending it

Easy wins that are already well-supported by the code:

- **different numbers of drones/defenders** — change `EnvConfig` (`n_drones`, `n_defenders`) and all wrappers auto-adapt to the new obs/action sizes.
- **different defender radius** — `defender_intercept_radius` in `configs/default.yaml`.
- **new route templates** — add a function to `envs/route_templates.py` and append to `ROUTE_TEMPLATES`.
- **polygon city** — extend `FixedMap.city_radius` into a polygon with a point-in-polygon check inside `dynamics.step_drone`.
- **mp4 replay** — pass `--save_path replay.mp4` to `animate_episode` (requires `ffmpeg`).

## Training expectations (prototype caveats)

The scripted heuristics contain real domain knowledge (nearest-threat
assignment, shortest-path routing, angular coverage). In short runs, a
PPO agent trained from scratch will **not** immediately match them — it
takes a lot of samples for a 128x128 MLP with `MultiDiscrete([6]*5)`
actions to learn "route to the threat along the road graph" from a
scalar reward signal.

What you should actually see:

- **Phase 2 (tactical defender)**: mean episode reward rises slowly. With
  ~200k steps the trained agent approaches the scripted baseline; with
  ~1M+ steps it can overtake it.
- **Phase 3 (attacker)**: this converges much faster, because the action
  is a one-shot plan and each RL step is a full episode. 30k-60k steps is
  usually enough to learn stronger launch-time / route template choices
  than random.
- **Phase 4 (deployment)**: also fast (one-shot).

If you want a satisfying training curve out of the box, the attacker and
deployment one-shot phases are the most rewarding to demo.
